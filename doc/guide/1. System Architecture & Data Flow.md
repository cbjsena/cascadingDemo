# **1\. 시스템 아키텍처 및 데이터 흐름도 (System Architecture & Data Flow)**

## **1\. 개요 (Overview)**

본 문서는 \*\*해운 물류 시나리오 시뮬레이션 시스템(Cascading Opt)\*\*의 전체 아키텍처와 데이터 흐름을 정의합니다.

본 시스템은 선박 스케줄(Proforma, LRS), 비용, 제약 조건 등 방대한 데이터를 관리하고, 선박 배치(Cascading)를 통해 장기 운항 계획(Long Range Schedule)을 생성한 뒤 최적화 엔진에 전달하여 결과를 도출하는 **순환형 시뮬레이션 플랫폼**입니다.

## **2\. 시스템 아키텍처 (System Architecture)**

시스템은 철저한 **관심사 분리(Separation of Concerns)** 원칙에 따라 설계되었습니다. 컨트롤러(View)는 요청/응답과 UI 렌더링만 담당하며, 모든 비즈니스 로직은 서비스(Service) 레이어로 캡슐화되었습니다. 대용량 데이터 조회를 위해 프론트엔드에 DataTables Server-side Rendering을 도입했습니다.

## **3\. 데이터 흐름 및 생명주기 (Data Lifecycle)**

데이터는 \*\*마스터/기준 정보(Base)\*\*에서 시작하여 \*\*시나리오(Scenario)\*\*로 파생되고, 사용자의 \*\*스케줄링(Scheduling)\*\*을 거쳐 엔진으로 전달되는 5단계 생명주기를 가집니다.

### **Phase 1\. 초기화 (Initialization)**

* **Source:** CSV Files (input\_data/data/base\_data/\*.csv)  
* **Process:** post\_migrate 시그널에 의한 자동 로드 (apps.py \-\> init\_base\_data, init\_master\_data)  
* **Key Logic:** 마스터 데이터가 이미 존재하는지 검사(exists())하여 최초 1회만 안전하게 데이터를 적재합니다. Lazy Loading 캐시(\_fk\_cache)와 bulk\_create를 사용하여 대량 데이터(예: Bunker Consumption Sea)의 적재 성능을 극대화했습니다.

### **Phase 2\. 시나리오 생성 (Scenario Creation)**

* **Source:** Base\_ Tables \-\> Sce\_ Tables  
* **Process:** create\_default\_scenario 커맨드 또는 UI의 대시보드 복제 요청  
* **Key Logic:** \* ScenarioInfo 생성 후 Base 데이터를 Scenario 데이터로 대량 복사합니다.  
  * **\[중요\]** Flat 구조였던 Base Proforma 데이터를 복제 시 **Master(헤더)와 Detail(기항지) 구조로 분해하여 정규화**합니다. (\_copy\_proforma\_to\_scenario)

### **Phase 3\. 스케줄링 및 배선 (Scheduling & Cascading) \[핵심 비즈니스\]**

* **Input:** UI를 통한 사용자의 데이터 입력 및 조작  
* **Process:**  
  1. **Proforma Schedule:** ProformaService가 포트 간 거리, 해상 시간(Sea Time), 속도(Speed), 입출항 시간(ETB/ETD)을 자동 역산하며, 역방향 날짜 입력 시 다음 주차로 자동 보정하는 방어 로직을 수행합니다.  
  2. **Cascading Vessel Position:** CascadingService가 Proforma의 기간(Duration)을 바탕으로 배선 슬롯의 날짜를 계산하고 선박을 배치합니다.  
  3. **Long Range Schedule (LRS) Generation:** LongRangeService가 시나리오 기간 동안 항차(Voyage)를 돌리며 스케줄을 무한 확장합니다. 이때 Turning Port(Y) 조건에 따라 \*\*가상 포트(Virtual Port)\*\*를 이전/현재 항차로 지능적으로 분리 생성합니다.

### **Phase 4\. 시뮬레이션 실행 (Simulation Execution) \[Planned\]**

* **Input:** 완성된 LRS 및 Cost, Vessel, Bunker, Constraint 데이터  
* **Process:** Celery Async Task를 통해 최적화 엔진에 데이터를 주입(JSON/CSV)하고 실행합니다.

### **Phase 5\. 결과 분석 (Result Analysis) \[Planned\]**

* **Output:** Engine Result (최적화된 스케줄 및 비용 데이터)  
* **Action:** Dashboard Visualization. 원본 시나리오(Sce\_)의 KPI(Total TEU, Utilization 등)와 엔진 최적화 결과를 템플릿 레이어에서 비교 분석합니다.

## **4\. 데이터베이스 모델링 구조 (Data Modeling Strategy)**

효율적인 데이터 관리와 확장성을 위해 **Abstract Base Class (추상 모델)** 패턴을 적용했습니다.

### **4.1 3단계 테이블 상속 구조**

모든 비즈니스 로직 테이블은 3단계 상속 구조를 가집니다.

1. **Abstract Model (Abs...)** : 실제 DB 테이블을 생성하지 않으며, 필드(컬럼) 정의만 담당합니다.  
2. **Base Model (Base...)** : 기준 정보(Master Data)를 저장하는 테이블입니다. (base\_ 접두어)  
3. **Scenario Model (Prefix 없음)** : ScenarioInfo FK를 가지며, 시뮬레이션용 개별 데이터를 저장합니다. (sce\_ 접두어)

### **4.2 주요 데이터 그룹 (Domain Groups)**

* **Schedule:** \* ProformaSchedule (Master) / ProformaScheduleDetail (기항지 상세)  
  * CascadingSchedule, CascadingVesselPosition (선박 배치)  
  * LongRangeSchedule (장기 운항 계획)  
  * LaneProformaMapping (Lane별 적용 Proforma 기간 매핑)  
* **Vessel:** VesselInfo (선박 제원), CharterCost (용선료), VesselCapacity (선복량)  
* **Cost & Distance:** Distance (운항 거리), CanalFee (운하 통행료), TSCost (환적 비용)  
* **Bunker:** BunkerConsumptionSea / BunkerConsumptionPort (연료 소모량), BunkerPrice (연료 단가)  
* **Constraint:** FixedVesselDeployment (선박 투입 제약), FixedScheduleChange (이벤트/수리 제약), PortConstraint (항만 제약)

## **5\. 자동화 및 운영 프로세스 (Automation & Ops)**

시스템 유지보수 효율성과 개발 편의성을 높이기 위해 주요 관리 작업이 자동화되어 있습니다.

* **DB Comment Automation:** apps.py의 post\_migrate 시그널을 통해, 모델의 verbose\_name을 실제 PostgreSQL DB 테이블 및 컬럼의 COMMENT로 자동 주입합니다. (DB 툴에서 즉시 의미 파악 가능)  
* **Table Definition Generation:** 마이그레이션 수행 시, 최신 스키마 정보를 반영한 \*\*테이블 정의서(CSV)\*\*가 doc/db/ 경로에 자동 갱신됩니다.  
* **Superuser Creation:** 마이그레이션 후 관리자 계정이 없으면 시스템 유저를 자동 생성합니다.  
* **Base Data Load:** input\_data/data/base\_data 경로의 CSV 파일들을 읽어 base\_ 접두어가 붙은 테이블에 자동 적재합니다. 데이터 용량에 따라 Bulk Insert 크기를 능동적으로 조절합니다.  
* **Test Isolation:** pytest 실행 시 위 자동화 로직을 모두 스킵(Skip)하도록 분기 처리하여, 테스트 환경의 속도와 안정성을 확보했습니다.
