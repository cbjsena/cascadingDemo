# **3\. 기능 명세서 (Input Data Module Specification)**

## **1\. 개요 (Overview)**

Input Data 모듈은 해운 물류 시뮬레이션의 기초가 되는 데이터를 적재(Load), 생성(Create), 편집(Edit)하는 핵심 도메인 앱입니다.

사용자는 불변하는 기준 정보(Master/Base Data)를 바탕으로 \*\*시나리오(Scenario)\*\*를 파생시키고, 웹 인터페이스를 통해 운항 스케줄(Proforma), 선박 배치(Cascading), 장기 운항 계획(LRS), 그리고 각종 비용 및 선박 제원을 정밀하게 조정할 수 있습니다.

## **2\. 데이터 초기화 및 시스템 자동화 (Data Initialization & Automation)**

management/commands 하위의 스크립트와 apps.py의 post\_migrate 시그널을 통해 데이터베이스 초기 구축 및 문서화를 자동화합니다.

### **2.1 마스터 및 기준 데이터 적재 (\_base\_loader.py)**

CSV 파일을 읽어 DB에 저장할 때 공통 로직을 처리합니다.

* **init\_master\_data**: master\_ 테이블 데이터를 적재합니다. 실행 시 기존 시나리오(sce\_)와 기준(base\_) 데이터가 연쇄 삭제(CASCADE)될 수 있음을 강력히 경고하고 방어 로직(--force 옵션)을 제공합니다.  
* **init\_base\_data**: base\_ 테이블 데이터만 안전하게 교체 적재합니다.  
* **최적화 로직**: tqdm 진행률 표시, Lazy Loading 방식의 FK 캐싱(\_fk\_cache), 그리고 대량 데이터(예: Bunker Consumption)에 대한 bulk\_create (배치 사이즈 5000)를 적용하여 적재 속도를 극대화했습니다.

### **2.2 DB 코멘트 및 명세서 자동화 (apps.py)**
마이그레이션 직후(post_migrate) 아래의 작업들이 자동 수행됩니다. 단, 테스트 환경(pytest)에서는 데이터 충돌 방지를 위해 시스템이 이를 감지하고 자동 비활성화합니다.
* **DB 코멘트 현행화**: 마이그레이션 직후 Django Model의 verbose\_name을 PostgreSQL COMMENT ON TABLE/COLUMN 쿼리로 자동 반영하여 DBeaver 등 외부 툴에서의 가독성을 높입니다.  
* **테이블 정의서 생성**: DB 시스템 카탈로그를 조회하여 최신 스키마 정보를 CSV 파일(base\_table\_definitions.csv)로 추출 및 자동 갱신합니다.

초기 관리자 생성: 시스템 구동 시 cascading (슈퍼유저) 계정을 자동 생성 및 부여합니다.

## **3\. 시나리오 관리 (Scenario Management)**

시뮬레이션의 실행 단위인 \*\*'시나리오(Scenario)'\*\*의 생명주기를 관리합니다.

### **3.1 시나리오 생성 및 복제 (scenario\_service.py)**

* **Base Scenario 생성**: 초기 마스터 데이터를 복제하여 기준 시나리오(BASELINE)를 생성합니다. UI 대시보드에서 AJAX(Fetch API)를 통해 즉시 호출(create_base_scenario_view)할 수 있습니다. 
* **시나리오 복제(Clone)**: 기존 시나리오의 모든 하위 데이터를 새로운 시나리오 ID로 통째로 복사합니다.  
  * **\[핵심 로직\]**: Flat 구조로 된 Base Proforma 데이터를 복사할 때, **Master(헤더)와 Detail(기항지) 구조로 분해(\_copy\_proforma\_to\_scenario)하여 정규화**한 뒤 bulk\_create로 고속 삽입합니다.  
  * **교차 참조 복제**: Proforma가 복제된 후, 발급된 새로운 Proforma ID를 CascadingVesselPosition 등의 연관 테이블 FK에 정확히 재매핑(\_clone\_relation\_data)합니다.
* **코드 자동 채번**: 시나리오 생성 시 SCYYYYMMDD_NNN 형태의 코드가 자동 증분되며, 동시성 충돌 발생 시 트랜잭션 롤백 후 재시도하는 방어 로직이 적용되어 있습니다.

### **3.2 대시보드 (scenario\_dashboard.html)**

시나리오별 노선 수(Total Lanes), 필요 선박 수(Required Vessels), 총 선복량(Total Capacity) 등의 핵심 KPI를 집계하여 직관적인 대시보드 형태로 제공합니다.

## **4\. 스케줄링 및 배선 (Scheduling & Cascading) \[★핵심 기능\]**

최적화 엔진에 전달할 최종 스케줄(LRS)을 만들기 위한 3단계 프로세스를 제공합니다.

### **Step 1\. Proforma Schedule 관리 (proforma\_service.py)**

노선(Lane)별 표준 스케줄 패턴(기항지 순서, 소요 시간 등)을 정의합니다.

* **Dynamic Grid UI**: 엑셀과 유사한 동적 웹 그리드를 제공하여 행 추가/삽입/삭제가 가능합니다. 입력 불가(Read-only) 셀은 시각적으로 비활성화됩니다.  
* **자동 역산 엔진**: ETB(접안) 및 Work Hours, Pilot In/Out 시간을 입력하면, **거리(Distance API 연동) \-\> 해상 시간(Sea Time) \-\> 속도(Speed) \-\> ETD(출항)를 자동으로 연쇄 계산**합니다.  
* **날짜 역전 방어**: 사용자가 입력한 ETB가 이전 항구의 ETD보다 과거일 경우(etb\_no 검증), 에러 처리 및 요일 기반 가장 가까운 미래 날짜로 자동 보정(\_resolve\_next\_etb)합니다.  
* **Excel I/O**: ExcelManager와 csv\_configs를 활용하여 엑셀 업로드/다운로드 기능을 완벽히 지원합니다.

### **Step 2\. Lane Proforma Mapping & Cascading (cascading\_service.py)**

표준 스케줄(Proforma)에 실제 선박을 배치하는 기능입니다.

* **Lane Proforma Mapping**: 특정 노선에 시기별로 적용할 Proforma를 선택합니다. UI에서는 가로형 타임라인(Timeline bar)을 통해 적용 기간(Effective Date)과 주차(Week)를 직관적으로 시각화합니다.  
* **Cascading Vessel Assignment**: Proforma에 정의된 필요 선박 수(declared\_count)만큼 슬롯(Position)을 생성하고, 각 슬롯의 투입일(vessel\_position\_date)을 Proforma의 기간(Duration)을 바탕으로 자동 계산합니다. 사용자는 각 슬롯에 실제 선박(vessel\_code)을 지정합니다.

### **Step 3\. Long Range Schedule (LRS) 생성 (long\_range\_service.py)**

Proforma와 Cascading 배선 결과를 조합하여 시뮬레이션 기간(planning\_horizon\_months) 전체의 장기 일정을 전개합니다.

* **Virtual Port (가상 포트) 파생 로직**: Proforma의 Turn Port Indicator가 Y인 경우, 시퀀스의 위치(Head Y, Tail Y 등)를 분석하여 이전 항차(Prev Voyage, offset: -1) 또는 현재 항차(Curr Voyage, offset: 0)의 반대 방향(Opposite Direction)으로 가상 기항지를 자동 파생(\_get\_expanded\_sequence)시킵니다.  
* **무한 루프 전개**: 선박의 1항차 시작일을 기준으로, 시나리오 종료일(lrs\_end\_date)에 도달할 때까지 Round Trip Time(Duration)을 더해가며 Voyage 번호를 증가시키고 스케줄을 찍어냅니다.

## **5\. 자원 및 제약 조건 관리 (Resources & Constraints)**

비용, 연료, 선박 제원 등을 관리하는 서브 화면들입니다. 이 화면들은 유지보수성을 극대화하기 위해 \*\*공통 팩토리 패턴(\_crud\_base.py)\*\*으로 구현되었습니다.

### **5.1 공통 CRUD UI / UX 아키텍처 (\_crud\_base.py)**
* **시나리오 팩토리 (_crud_base.py)**: scenario_crud_view를 통해 특정 시나리오(scenario_id)에 종속된 데이터(Cost, Bunker, Vessel 등)의 목록 조회, 모달 추가, 일괄 삭제, CSV I/O를 자동 생성합니다.
* **마스터 팩토리 (master.py)**: master_crud_view를 통해 시나리오와 무관한 전사 공통 불변 데이터(Trade, Port 등)의 CRUD를 자동 생성합니다.
* **Server-side DataTables**: 대용량 데이터를 브라우저에서 버벅임 없이 처리하기 위해 AJAX 기반의 서버사이드 페이징, 정렬, 검색을 지원합니다.  (_datatables_base.html / master_list_base.html 상속)
* **모달(Modal) 기반 편집**: 화면 이동 없이 Add Row 버튼을 통해 데이터를 추가합니다.  
* **Upsert 및 중복 방어**: 설정(unique\_fields, lookup\_fields)에 따라 update\_or\_create 또는 중복 Skip을 자동 수행합니다.  
* **일괄 CSV 연동**: 화면에 표시된 필터 상태 그대로 CSV 다운로드 및 대량 업로드를 지원합니다.

### **5.2 주요 관리 도메인(Scenario Level)**

1. **Cost & Distance (cost.py)**  
   * **Distance**: 두 항구 간 운항 거리 (ECA 거리 포함). 자동 계산 엔진에서 즉시 참조됩니다.  
   * **Canal Fee / TS Cost**: 운하 통행료 및 환적 비용 관리.  
2. **Bunker (bunker.py)**  
   * **Bunker Consumption**: 선박 크기(TEU) 및 선속(Speed 0.5단위)에 따른 해상 소모량 테이블과 정박/아이들링 소모량 관리.  
   * **Bunker Price**: 연료 타입(LSFO, MGO) 및 지역/노선별 단가 관리.  
3. **Vessel (vessel.py)**  
   * **Vessel Info**: 커스텀 로직이 적용된 선박 마스터 정보 (Dry Dock 수리 일정, 인도/반선 정보 등).  
   * **Charter Cost / Capacity**: 선박별/기간별 용선료 및 적재 가능량(TEU) 정보 관리.

### **5.3 마스터 데이터 관리 (Master Level)**

모든 시나리오에서 공통으로 참조하는(ON_DELETE=PROTECT) 핵심 기준 정보입니다.
1. **Master Trade / Port / Lane**: 해운 물류의 가장 뼈대가 되는 Location 및 Service 코드 관리 
2. **Week Period (base_week_period)**:특정 연도/주차(YYYYWK)가 실제 달력의 어느 시작일~종료일에 해당하는지 맵핑하는 주차 마스터 정보

## **6\. 기능 명세 검증 및 BDD (Behavior-Driven Development)**

본 명세서의 모든 기능 요구사항은 test_scenarios_input_data.csv, test_scenarios_api.csv 파일에 정의된 190여 개의 BDD(Given-When-Then) 기반 테스트 시나리오와 1:1로 맵핑됩니다.
* **명세의 코드화**: 정의된 시나리오 ID (예: INPUT_SCENARIO_CREATE_001, LRS_SVC_001, MASTER_TRADE_001)는 tests/ 폴더 하위의 pytest 코드에 그대로 투영되어, 시스템의 모든 기능이 기획 명세와 정확히 일치하게 동작함을 자동화된 테스트로 엄격하게 보장합니다.
