# **3. 입력 데이터 모듈 명세서 (Input Data Module Specification)**

## **1. 개요 (Overview)**

Input Data 모듈은 해운 물류 시뮬레이션의 기초가 되는 데이터를 적재(Load), 생성(Create), 편집(Edit)하는 핵심 도메인 앱입니다.

사용자는 불변하는 기준 정보(Master/Base Data)를 바탕으로 **시나리오(Scenario)**를 파생시키고, 웹 인터페이스를 통해 운항 스케줄(Proforma), 선박 배치(Cascading), 장기 운항 계획(LRS), 그리고 각종 비용 및 선박 제원을 정밀하게 조정할 수 있습니다.

### **1.1 모듈 구성**
```
input_data/
├── views/              # 사용자 인터페이스 (MVC Controller)
│   ├── scenario.py    (시나리오 관리)
│   ├── proforma.py    (Proforma 스케줄)
│   ├── cascading.py   (선박 배치)
│   ├── lane_proforma.py (노선-Proforma 매핑)
│   ├── long_range.py  (LRS 생성)
│   ├── cost.py        (비용 관리)
│   ├── bunker.py      (연료 관리)
│   ├── vessel.py      (선박 관리)
│   ├── _crud_base.py  (공통 CRUD 팩토리)
│   └── _datatables_base.html (DataTables 기반 Grid UI)
├── services/          # 비즈니스 로직 (Fat Service)
│   ├── proforma_service.py

│   ├── long_range_service.py
```

## **2. 데이터 초기화 및 시스템 자동화 (Data Initialization & Automation)**

management/commands 하위의 스크립트와 apps.py의 post_migrate 시그널을 통해 데이터베이스 초기 구축 및 문서화를 자동화합니다.

### **2.1 마스터 및 기준 데이터 적재 (_base_loader.py)**

CSV 파일을 읽어 DB에 저장할 때 공통 로직을 처리합니다.

* **init_master_data**: master_ 테이블 데이터를 적재합니다. 
  * 실행 시 기존 시나리오(sce_)와 기준(base_) 데이터가 연쇄 삭제(CASCADE)될 수 있음을 강력히 경고
  * 방어 로직: --force 옵션 없으면 실행 안 됨 (안전 확인 프롬프트)
  * 용도: 시스템 초기 구성 또는 전체 마스터 데이터 재초기화 시에만 사용

* **init_base_data**: base_ 테이블 데이터만 안전하게 교체 적재합니다.
  * base_ 데이터 갱신 중에는 sce_ 데이터는 영향받지 않음
  * 용도: 선박 정보, 연료비, 거리 데이터 등 기준 정보 업데이트 시 사용

* **create_default_scenario**: 초기 기준 시나리오(BASELINE) 자동 생성
  * Base Data를 복제하여 첫 번째 시나리오 생성
  * 관리자가 수동으로 생성하거나 커맨드로 자동 생성 가능

* **최적화 로직**:
  * **tqdm 진행률 표시**: 대용량 데이터 적재 시 실시간 진행 상황 시각화
  * **Lazy Loading FK 캐싱**: _fk_cache를 통해 반복 쿼리 최소화
  * **bulk_create 배치**: 배치 크기 5000으로 대량 데이터(예: Bunker Consumption) 고속 삽입
  * **ignore_conflicts**: 중복 데이터 에러 무시하고 계속 진행

### **2.2 DB 코멘트 및 명세서 자동화 (apps.py)**

마이그레이션 직후(post_migrate) 아래의 작업들이 자동 수행됩니다. 단, 테스트 환경(pytest)에서는 데이터 충돌 방지를 위해 시스템이 이를 감지하고 자동 비활성화합니다.

* **update_db_comments**: Django Model의 verbose_name을 PostgreSQL COMMENT ON TABLE/COLUMN 쿼리로 자동 반영
  * 효과: DBeaver 등 외부 DB 도구에서 스키마 컬럼의 의미를 즉시 파악 가능
  * 구현: apps.py의 post_migrate 시그널에서 모든 모델 순회 및 SQL 실행

* **generate_table_definition**: DB 시스템 카탈로그를 조회하여 최신 스키마 정보를 CSV 파일로 추출
  * 파일 위치: `doc/db/base_table_definitions.csv`
  * 포함 정보: 테이블명, 컬럼명, 데이터 타입, 코멘트, 제약 조건
  * 자동 갱신: 마이그레이션마다 최신 상태 반영

* **create_default_superuser**: 시스템 초기 구동 시 관리자 계정 자동 생성
  * 기본 계정: cascading / qwer123$
  * 기존 계정 존재 시: 생성 스킵
  * 수정 가능: 이후 Django Admin에서 비밀번호 변경 가능

## **3. 시나리오 관리 (Scenario Management)**

시뮬레이션의 실행 단위인 **'시나리오(Scenario)'**의 생명주기를 관리합니다.

### **3.1 시나리오 생성 및 복제 (scenario_service.py / scenario.py)**

#### **Base Scenario 생성**
* **초기 생성**: 초기 마스터 데이터를 복제하여 기준 시나리오(BASELINE)를 생성
* **UI 호출**: 대시보드의 "Create Default Scenario" 버튼 클릭 시 AJAX(Fetch API)를 통해 즉시 호출
* **자동 코드 채번**: SCYYYYMMDD_NNN 형식으로 자동 증분
  * SC: 고정 접두어 (SC = Scenario)
  * YYYYMMDD: 생성 날짜
  * NNN: 일일 일련번호 (001, 002, ...)
  * 동시성 안전: IntegrityError 발생 시 최대 10회 재시도 (transaction.atomic)

#### **시나리오 복제(Clone)**
* **용도**: 기존 시나리오의 모든 하위 데이터를 새로운 시나리오 ID로 통째로 복사
* **복제 대상**: ScenarioInfo, ProformaSchedule, Cascading, LRS, Cost, Bunker, Vessel 등 모든 sce_ 테이블 데이터
* **핵심 로직 - Proforma 정규화**:
  ```
  Flat 구조 (Base):
    Base Proforma (1개 행) → 여러 기항지가 같은 행에 섞임
  
  Normalized 구조 (Scenario):
    Proforma Schedule (Master): 1개 행 → 헤더 정보 (lane, proforma_name, duration 등)
    Proforma Schedule Detail: N개 행 → 각 기항지별 상세 (port_seq, etb, etd 등)
  ```
  * 구현: scenario_service.py의 _copy_proforma_to_scenario 함수

* **교차 참조 복제**:
  - Proforma가 복제된 후, 발급된 새로운 Proforma ID를 자동으로 추적
  - CascadingSchedule, CascadingVesselPosition 등 하위 FK 참조를 정확히 재매핑
  - 구현: _clone_relation_data 함수에서 ID 매핑 딕셔너리 유지

* **성능 최적화**:
  - bulk_create 활용: 대량 INSERT를 일괄 처리
  - transaction.atomic: 전체 복제 과정을 원자적(Atomic) 트랜잭션으로 처리
  - 부분 실패 시 전체 롤백으로 데이터 일관성 보장

#### **메타데이터 관리**
* `base_scenario`: 복제 원본 시나리오 추적 (NULL이면 원본)
* `scenario_type`: BASELINE, WHAT_IF, OPTIMIZATION, SENSITIVITY, COMPARISON
* `scenario_status`: DRAFT (작성 중), ACTIVE (분석 중), ARCHIVED (완료), BASELINE (기준)
* `base_year_week`: 시뮬레이션 시작 주차 (YYYYWK 형식, 예: 202610)
* `planning_horizon_months`: 계획 기간 (개월, 예: 12)
* `to_year_week`: 계산된 종료 주차 (property로 자동 계산)
* `tags`: 사용자 정의 태그 (콤마 구분, 예: "LNG, Asia-Route")
* `calculation_status`: PENDING, PROCESSING, COMPLETED, ERROR

### **3.2 시나리오 대시보드 (scenario_list.html)**

시나리오별 핵심 메트릭을 한눈에 집계하여 직관적인 대시보드 형태로 제공합니다.

* **주요 KPI 집계**:
  - **Total Lanes**: 시나리오에 포함된 노선의 개수
  - **Required Vessels**: 모든 Proforma의 declared_count 합계 (필요 선박 수)
  - **Total Capacity (TEU)**: 모든 선박의 선복량 합계
  - **Calculation Status**: 현재 계산 진행 상태 (진행률 표시)

* **시나리오 목록 테이블**:
  - 시나리오 코드, 유형, 상태, 생성일, 기간 등 정보 표시
  - 각 시나리오별 Action 버튼 (Edit, Clone, Export, Delete)
  - Server-side DataTables로 대용량 시나리오 처리

* **빠른 생성 UI**:
  - "New Scenario" 버튼 → 모달 팝업 (형식, 기간, 원본 선택)
  - "Clone" 버튼 → 선택된 시나리오를 기반으로 복제 시작

## **4. 스케줄링 및 배선 (Scheduling & Cascading) [★핵심 기능]**

최적화 엔진에 전달할 최종 스케줄(LRS)을 만들기 위한 3단계 프로세스를 제공합니다.

### **Step 1. Proforma Schedule 관리 (proforma_service.py / proforma.py)**

노선(Lane)별 표준 스케줄 패턴(기항지 순서, 소요 시간 등)을 정의합니다.

#### **Dynamic Grid UI (proforma.html)**
* **엑셀 유사 인터페이스**: 엑셀과 유사한 동적 웹 그리드 제공
  * 행 추가/삽입/삭제 가능 (Add Row, Insert Row, Delete Row 버튼)
  * 입력 불가(Read-only) 셀은 시각적으로 비활성화 (회색 처리)
  * Keyboard Navigation: Tab 키로 셀 간 이동, Enter로 행 이동

* **액션 버튼**:
  - **New**: 새로운 Proforma 작성 시작 (화면 초기화)
  - **Load**: 기존 Proforma 조회 (시나리오/노선/Proforma 선택)
  - **Calculate**: 일부 수정 후 재계산 수행
  - **Save**: 최종 저장 (DB에 Master & Detail 정규화 저장)
  - **Download**: 현재 데이터를 Excel로 다운로드
  - **Upload**: Excel 파일로 대량 일괄 업로드

#### **자동 역산 엔진 (ProformaService.calculate_schedule)**
사용자가 특정 필드를 입력하면, 나머지 필드가 자동으로 계산되는 스마트 엔진입니다.

```
입력 정보:
  1. Port Code → Distance API로 거리 조회
  2. Work Hours → 항만 작업 시간
  3. Pilot In/Out Hours → 외항에서 부두까지 소요 시간

자동 계산:
  1. Sea Time = Distance / Average Speed
  2. ETD(출항) = ETB(이전항) + Work Hours + Port Time (자동 역산)
  3. Next Port ETB = ETD + Sea Time (자동 전개)
  4. Speed = Distance / Sea Time (필요 시 역산)
```

**계산 순서 (예시)**:
```
Port A (ETB: 2026-01-01 09:00)
  ↓ [Work: 36h]
  ETD: 2026-01-02 21:00
  ↓ [Distance: 720nm, Speed: 20kn → Sea Time: 36h]
  
Port B (ETB: 2026-01-04 09:00)
  ↓ [Work: 24h]
  ETD: 2026-01-05 09:00
  ↓ [Distance: 480nm, Speed: 20kn → Sea Time: 24h]
  
Port C (ETB: 2026-01-06 09:00)
```

#### **날짜 역전 방어 로직 (_resolve_next_etb)**
사용자가 입력한 ETB가 이전 항구의 ETD보다 과거일 경우 자동 보정합니다.

```
검증 규칙:
  - Current ETB >= Previous ETD + Sea Time 이어야 함
  
자동 보정 로직:
  - 입력된 날짜가 조건을 만족하지 않으면
  - 해당 요일의 가장 가까운 미래 날짜로 자동 조정
  - 예: "월요일 09:00"이 조건 불만족 시 → "다음 월요일 09:00"으로 보정
  
에러 처리:
  - 사용자에게 보정 사실 명시 (경고 메시지)
  - 여전히 조건 불만족 시 저장 거부 (ValidationError)
```

#### **Excel I/O (ExcelManager)**
* **다운로드**: 현재 작성 중인 Proforma를 Excel로 다운로드
  - Template 제공: 컬럼 순서, 데이터 검증 규칙 포함
  - Conditional Formatting: 계산된 셀 vs 사용자 입력 셀 시각적 구분

* **업로드**: Excel 파일에서 다량의 Proforma 행을 읽어 일괄 삽입
  - Header Detection: 첫 행이 컬럼명인지 자동 감지
  - Validation: 각 행의 데이터 타입 및 필수 필드 검증
  - 부분 성공: 일부 행 오류 시에도 정상 행은 계속 진행 (rollback 아님)

#### **데이터 정규화 (_normalize_flat_to_master_detail)**
사용자 관점에서는 1개의 Grid로 보이지만, DB 저장 시에는 정규화된 구조로 저장됩니다.

```
저장 전 (입력 상태):
┌─────────────────────────────────────────────────────────────────────┐
│ Lane │ Name │ Duration │ Port1 │ Port2 │ Port3 │ Port4 │ Port5 │...│
├─────┼──────┼──────────┼───────┼───────┼───────┼───────┼───────┼───┤
│ AEX  │ P001 │   21    │ KRPUS │ CNSHA │ THLCH │ SGSIN │ KRPUS │...│
└─────────────────────────────────────────────────────────────────────┘

저장 후 (정규화된 구조):

proforma_schedule (Master):
┌────┬────────┬─────────────┬──────────┬────────┐
│ id │ lane   │ proforma    │ duration │ ...    │
├────┼────────┼─────────────┼──────────┼────────┤
│ 42 │ AEX    │ P001        │    21    │        │
└────┴────────┴─────────────┴──────────┴────────┘

proforma_schedule_detail (Detail):
┌────┬──────────┬───────┬────────┬─────┬─────┬───┐
│ id │proforma │ port  │port_seq│ etb │ etd │...│
├────┼──────────┼───────┼────────┼─────┼─────┼───┤
│ 1  │    42    │KRPUS │   1    │ ... │ ... │   │
│ 2  │    42    │CNSHA │   2    │ ... │ ... │   │
│ 3  │    42    │THLCH │   3    │ ... │ ... │   │
│ 4  │    42    │SGSIN │   4    │ ... │ ... │   │
│ 5  │    42    │KRPUS │   5    │ ... │ ... │   │
└────┴──────────┴───────┴────────┴─────┴─────┴───┘
```

**이점**:
  - Master: 스케줄의 기본 정보 (하나의 행)
  - Detail: 각 기항지의 상세 정보 (여러 행)
  - 정규화로 인한 메모리 효율성 및 쿼리 성능 향상
  - Turning Port 분석 및 Virtual Port 생성에 용이

### **Step 2. Lane Proforma Mapping & Cascading (lane_proforma.py / cascading.py)**

표준 스케줄(Proforma)에 실제 선박을 배치하는 기능입니다.

#### **Lane Proforma Mapping**
* **목적**: 특정 노선에 시기별로 어떤 Proforma를 적용할지 매핑
* **UI 표현**: 가로형 Timeline Bar
  - X축: 시간축 (주차 단위)
  - Y축: 각 노선별 Proforma 타일
  - 기간 선택 가능: 드래그로 기간 조정 또는 Date Picker 사용

* **데이터 저장**: lane_proforma_mapping 테이블
  ```
  lane_code: "AEX"
  proforma_name: "P001"
  effective_from_date: 2026-01-01
  effective_to_date: 2026-06-30
  ```

* **용도**: 계절에 따라 다른 Proforma 적용 가능
  - 예: 1분기 P001, 2분기 P002 (성수기 다른 스케줄), 3분기 P001로 재전환

#### **Cascading Vessel Assignment**
* **목적**: Proforma에 정의된 필요 선박 수만큼 슬롯(Position)을 생성하고 선박 배치

* **프로세스**:
  1. **포지션 생성**: declared_count = 3이면 Position 1, 2, 3 자동 생성
  2. **투입일 자동 계산**: Proforma의 Duration을 바탕으로 각 포지션의 vessel_position_date 자동 계산
     ```
     Duration = 21일 (한 사이클 소요 시간)
     Position 1: 2026-01-01 출항
     Position 2: 2026-01-22 출항 (Position 1 + 21일)
     Position 3: 2026-02-12 출항 (Position 2 + 21일)
     ```
  3. **선박 지정**: 사용자가 각 Position에 실제 선박 코드 입력
  4. **소유 구분**: Own(자선) vs Chartered(용선) 지정

* **데이터 구조**:
  ```
  cascading_schedule:
    - scenario, proforma, vessel_position
  
  cascading_vessel_position:
    - cascading_schedule, vessel_code, vessel_position_date, own_type
  ```

### **Step 3. Long Range Schedule (LRS) 생성 (long_range_service.py)**

Proforma와 Cascading 배선 결과를 조합하여 시뮬레이션 기간(planning_horizon_months) 전체의 장기 일정을 전개합니다.

#### **Virtual Port (가상 포트) 파생 로직**
Proforma의 Turn Port Indicator가 Y인 경우, 시퀀스의 위치를 분석하여 가상 기항지를 자동 파생합니다.

```
Turn Port Y의 위치에 따라:

[경우 1] Head Y (시작 기항지가 Turning Port)
  - 이전 항차의 마지막 기항지 다음에 Virtual Port 삽입
  - 항차 전환점 표시

[경우 2] Tail Y (마지막 기항지가 Turning Port)
  - 현재 항차의 마지막 기항지 다음에 Virtual Port 삽입
  - 다음 항차 시작 전 전환점 표시

[경우 3] Middle Y (중간 기항지가 Turning Port)
  - 해당 기항지를 중심으로 시퀀스 분석
  - 이전 항차와 현재 항차의 경계점으로 설정

효과:
  - 최적화 엔진이 항차 경계를 정확히 인식 가능
  - 항차별 비용 및 제약 조건 분리 계산 가능
```

**구현**: _get_expanded_sequence 함수

#### **무한 루프 전개 (_generate_lrs_voyages)**
선박의 1항차 시작일을 기준으로, 시나리오 종료일(lrs_end_date)에 도달할 때까지 스케줄을 반복 생성합니다.

```
입력:
  - vessel: 선박 정보
  - start_date: 선박 투입 시작일 (2026-01-01)
  - lrs_end_date: 시나리오 종료일 (2026-12-31)
  - duration: Proforma 사이클 기간 (21일)

로직:
  voyage_num = 1
  current_etb = start_date
  
  while current_etb <= lrs_end_date:
      - Voyage 생성 (voyage_num, etb, etd, port, ...)
      - Turning Port 기반 Virtual Port 추가
      - current_etb += duration
      - voyage_num += 1

결과: 시나리오 기간 전체를 커버하는 상세 항차 리스트
```

**최적화**:
  - Bulk Create로 대량 voyage 일괄 INSERT
  - 메모리 효율을 위해 배치 단위(예: 1000개)로 나누어 처리

## **5. 자원 및 제약 조건 관리 (Resources & Constraints)**

### **5.1 공통 CRUD UI/UX 아키텍처 (_crud_base.py)**

#### **Scenario Factory Pattern (scenario_crud_view)**
특정 시나리오(scenario_id)에 종속된 데이터(Cost, Bunker, Vessel 등)의 CRUD를 자동 생성합니다.

* **자동 생성 기능**:
  - 목록 조회: 필터링, 정렬, 페이징
  - 추가: Modal 팝업을 통한 신규 행 삽입
  - 수정: 각 행의 inline 편집 또는 Modal 팝업
  - 삭제: 선택된 여러 행 일괄 삭제
  - 내보내기: CSV/Excel 형식 다운로드
  - 가져오기: CSV/Excel 파일 대량 업로드

* **설정 주입 (Configuration Injection)**:
  ```python
  # views/cost.py에서 설정 정의
  class DistanceCRUD(scenario_crud_view):
      model = CostDistance
      fields = ['from_port', 'to_port', 'distance_nm', 'eca_distance_nm']
      unique_fields = ['from_port', 'to_port']  # 중복 체크
      searchable_fields = ['from_port__port_name', 'to_port__port_name']
  ```

* **Server-side DataTables**:
  - AJAX 기반의 서버사이드 페이징, 정렬, 검색
  - 대용량 데이터(수백만 행)도 브라우저에서 버벅임 없이 처리
  - 템플릿: _datatables_base.html / master_list_base.html 상속

#### **Master Factory Pattern (master_crud_view)**
시나리오와 무관한 전사 공통 불변 데이터(Trade, Port, Lane)의 CRUD를 자동 생성합니다.

* **특징**:
  - Scenario FK가 없음 (Master 데이터는 공통)
  - 수정/삭제 시 참조 무결성 확인 (ON_DELETE=PROTECT)
  - 마스터 데이터 삭제 시 하위 sce_ 데이터 존재 여부 검사

#### **Modal 기반 편집 (Add/Edit Modal)**
화면 이동 없이 모달 팝업을 통해 데이터 추가/수정합니다.

* **Add Row 버튼**:
  - 클릭 시 빈 Modal 폼 팝업
  - 필수 필드 검증 (빨간 별 표시)
  - Save 버튼으로 AJAX 저장 (새로고침 없이)

* **Edit Row 버튼**:
  - 기존 데이터 로드하여 Modal 폼에 pre-fill
  - Upsert 로직: update_or_create 사용
  - 중복 시 업데이트, 없으면 신규 생성

#### **Upsert 및 중복 방어**
설정(unique_fields, lookup_fields)에 따라 UPDATE 또는 CREATE 자동 결정합니다.

```python
# 예시 1: unique_fields 지정 시
unique_fields = ['from_port', 'to_port']

# → 동일 (from_port, to_port) 조합 존재 시 UPDATE
# → 없으면 CREATE

# 예시 2: lookup_fields 지정 시
lookup_fields = ['vessel', 'start_date']

# → (vessel, start_date) 조합으로 조회하여
#   → 결과 있으면 UPDATE / 없으면 CREATE
```

#### **일괄 CSV 연동**
화면에 표시된 필터 상태 그대로 CSV 다운로드 및 대량 업로드를 지원합니다.

* **CSV 다운로드**:
  - 현재 페이지 또는 전체 데이터 선택 가능
  - 필터 상태 유지 (검색어, 정렬)
  - 헤더 행 포함

* **CSV 업로드**:
  - 헤더 자동 감지 (첫 행이 컬럼명)
  - 각 행의 데이터 타입 및 필수 필드 검증
  - 부분 성공: 일부 행 오류 시에도 정상 행은 계속 진행
  - 결과 리포트: 추가됨 N개, 업데이트됨 N개, 오류 N개 표시

### **5.2 주요 관리 도메인 (Scenario Level)**

#### **1. Cost & Distance (cost.py)**

**Distance (`cost_distance`)**:
* 필드: from_port, to_port, distance_nm, eca_distance_nm, average_speed_knots
* 용도: Proforma 자동 역산 엔진에서 즉시 참조

* 필드: canal_name, from_port, to_port, direction, fee_per_teu, fee_fixed

* 필드: ts_port, cost_per_teu, cost_per_container
* 환적 항만의 비용
* 예시:

**Bunker Consumption Sea (`bunker_consumption_sea`)**:
* 필드: vessel_size, speed_knots, consumption_ton_per_day

**Bunker Consumption Port (`bunker_consumption_port`)**:
* 필드: vessel_size, port_idle_consumption_ton_per_day
* 항만 정박 중 아이들링 소비량
* 예시:
  ```
  Panamax → 5 ton/day (정박)
  New Panamax → 8 ton/day (정박)
  ```

**Bunker Price (`bunker_price`)**:
* 필드: bunker_type(LSFO/MGO), region, effective_date, price_per_ton
* 연료 유형별, 지역별 단가 관리
* 예시:
  ```

  MGO, Rotterdam, 2026-01: 520 $/ton
  ```

#### **3. Vessel (vessel.py)**

**Vessel Info (`vessel_info`)**:
* 커스텀 로직이 적용된 선박 마스터 정보
* 필드: vessel_code, vessel_name, own_type(Own/Chartered), vessel_size, delivery_date, redelivery_date, vessel_status
* 예시:

* 필드: vessel, cost_type, start_date, end_date, hire_rate ($/day)
* 기간별 용선료 관리
* 예시:
  ```
  VESSEL001: Time Charter, 2026-01~06: $25,000/day
  VESSEL001: Time Charter, 2026-07~12: $28,000/day
  ```

**Vessel Capacity (`vessel_capacity`)**:
* 필드: vessel, lane, direction, capacity_teu
* 노선/방향별 실제 적재 가능량
* 예시:
  ```
  VESSEL001, AEX(East): 4800 TEU
  VESSEL001, AEX(West): 4700 TEU (기후 조건)
  ```

### **5.3 마스터 데이터 관리 (Master Level)**


**Week Period (`base_week_period`)**:
* 특정 연도/주차(YYYYWK)가 실제 달력의 어느 start_date ~ end_date에 해당하는지 매핑

* 예시:


```python
    setup_test_data(given)
    
    # When: 실행
    result = perform_action(when)
    
    # Then: 검증
    assert_expected(result, expected)
```
