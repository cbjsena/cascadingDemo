# 2. 데이터베이스 설계 및 모델링 가이드 (Database Design & Modeling Guide)

## 1. 개요 (Overview)
본 시스템은 시뮬레이션의 기준이 되는 **마스터 데이터(Base Data)**와 시뮬레이션별로 독립적으로 조작 가능한 **시나리오 데이터(Scenario Data)**를 엄격히 분리하여 관리합니다. 
Django의 `Abstract Base Class(추상 모델)` 패턴을 적용하여 데이터 구조의 일관성을 보장하며, 복잡한 해운 물류 데이터를 효율적으로 처리하기 위해 고도의 **정규화(Normalization)** 및 **계층화** 설계를 적용했습니다.

---

## 2. 데이터 모델링 원칙 (Modeling Principles)

### 2.1 3단계 추상 모델 패턴 (3-Tier Abstract Model Pattern)
핵심 비즈니스 로직 테이블은 중복 코드를 방지하기 위해 3단계 상속 구조를 가집니다.
1. **Abstract Model (`Abs...`)**: 실제 DB 테이블을 생성하지 않으며, 필드 정의만 담당합니다.
2. **Base Model (`Base...`)**: 기준 정보(Master Data)를 저장하는 테이블입니다. (Scenario FK 없음)
3. **Scenario Model (Prefix 없음)**: 특정 시나리오(`ScenarioInfo`)에 종속된 데이터 저장소입니다.

### 2.2 테이블 명명 규칙 (Naming Convention)

| 모델 클래스명 | DB 테이블명 | 역할 및 특징 |
| :--- | :--- | :--- |
| `AbsVesselInfo` | (None) | 필드 정의 (상속용 추상 클래스) |
| `BaseVesselInfo` | `base_vessel_info` | **[Master]** 전사 공통 기준 데이터 (Scenario FK 없음) |
| `VesselInfo` | `sce_vessel_info` | **[Input]** 시나리오별 시뮬레이션 입력 데이터 (Scenario FK 포함) |
| `SimulationRun` | `sce_simulation_run` | **[Result]** 시뮬레이션 실행 기록 및 파라미터 내역 |

---

## 3. 공통 모델 정의 (Common Models)
모든 테이블에 공통으로 적용되는 감사(Audit) 로그 및 관계 설정을 위한 부모 모델입니다.

### 3.1 CommonModel (`common/models.py`)
데이터의 생성/수정 이력을 추적하기 위한 추상 모델입니다.
* `created_at` / `updated_at`: 생성 및 수정 일시 (Auto Now)
* `created_by` / `updated_by`: 생성자 및 수정자 (User FK)

### 3.2 ScenarioBaseModel (`input_data/models.py`)
시나리오 파생 데이터(`Sce_`)가 반드시 가져야 할 외래 키(FK)를 정의합니다.
* `scenario`: `ScenarioInfo` 모델 참조 (`ON_DELETE=CASCADE`)
* **특징:** 부모 시나리오 삭제 시, 하위 수십 개 테이블의 데이터가 DB 레벨에서 즉시 연쇄 삭제(Cascade)되어 고아 데이터를 방지합니다.

## 4. 논리적 데이터 그룹 (Logical Data Groups)

### 4.1 Scenario Master (Root) & Analysis
모든 시나리오 데이터의 뿌리가 되는 메타 정보 및 분석 결과 그룹입니다.
* **ScenarioInfo (`sce_scenario_info`)**: `code` (SCYYYYMMDD_NNN 자동 생성 고유 코드), `base_scenario` (복제 원본 추적 FK), `base_year_week`, `planning_horizon_months` 관리.
* **SimulationRun (`sce_simulation_run`)**: 최적화 엔진 실행 이력, 파라미터 및 요약 결과.
* **KPISnapshot (`sce_kpi_snapshot`)**: 시나리오별 주요 지표(Total TEU, Total Vessels, Utilization 등) 이력 보관.
* **ScenarioComparison (`sce_scenario_comparison`)**: 여러 시나리오 간의 비교 분석 결과 맵핑.

### 4.2 Group 1: Schedule & Cascading [★핵심]
* **Proforma Schedule (`schedule_proforma` / `schedule_proforma_detail`)**
  * **Master:** 노선별 스케줄 헤더 정보 (기간, 선복량, 필요 선박 수 등).
  * **Detail:** 개별 기항지 순서, 입출항 시간(ETB/ETD), 소요 시간(Sea Time), 거리.
  * **특징:** Excel 업/다운로드 시 UI에서는 1개의 Grid로 보이지만, DB 저장 시(Service Layer) Header와 Detail로 완벽히 정규화되어 저장됩니다.
* **Lane Proforma Mapping (`lane_proforma_mapping`)**
  * 역할: 특정 노선(Lane)에 시기별로 어떤 Proforma를 적용할지 타임라인 기반으로 매핑.
* **Cascading Schedule (`schedule_cascading`)**
  * 역할: Proforma에서 선박을 투입할 대상 **포지션(Position)** 번호들을 미리 선택하고 확보하는 스케줄 데이터 (선박 코드 미지정).
* **Cascading Vessel Position (`schedule_cascading_vessel_position`)**
  * 역할: Cascading Schedule에서 확보한 **포지션(Position)**에 실제 선박(`vessel_code`)을 배치하고 투입일(`vessel_position_date`)을 매핑.
* **Long Range Schedule (`schedule_long_range`)**
  * 역할: 최적화 엔진의 최종 입력값이 되는 장기 운항 계획.
  * 로직: Proforma와 배선 데이터를 결합하여 항차(Voyage)를 무한 루프 돌리며, Turning Port(`Y`) 조건에 따라 **가상 포트(Virtual Port)**를 자동 파생 생성합니다.

### 4.3 Group 2: Vessel (선박)
* **Vessel Info (`vessel_info`)**: 선박 제원 (소유 구분, 인도/반선일, 수리 일정).
* **Charter Cost (`vessel_charter_cost`)**: 용선료 정보 (기간별 `hire_rate`).
* **Vessel Capacity (`vessel_capacity`)**: 항로/선박/방향별 적재 가능량(TEU).

### 4.4 Group 3: Cost & Distance (비용 및 거리)
* **Distance (`cost_distance`)**: 항구 간 표준 운항 거리 및 ECA(배출통제해역) 거리.
* **Canal Fee (`cost_canal_fee`)**: 운하(Suez, Panama 등) 통과 방향별 비용.
* **TS Cost (`cost_ts_cost`)**: 환적(Transshipment) 항만 비용.

### 4.5 Group 4: Bunker (연료)
* **Bunker Consumption (`bunker_consumption_sea` / `_port`)**: 선박 크기 및 선속(Speed)별 해상 소모량 및 정박/아이들링 소모량.
* **Bunker Price (`bunker_price`)**: 연료 타입(LSFO, MGO) 및 지역/노선별 단가.

### 4.6 Group 5: Constraints (제약 조건)
* **Fixed Deployment (`constraint_fixed_deployment`)**: 항로 선박 투입 필수/금지 제약.
* **Fixed Schedule Change (`constraint_fixed_schedule_change`)**: Phase-In/Out 등 고정 이벤트.
* **Port Constraint (`constraint_port`)**: 항구별 입항 가능 선박 크기 제한.

### 4.7 Group 6: Master Data (불변 기준 정보)
* **MasterTrade, MasterPort, MasterLane**: 전사 공통으로 쓰이는 절대 기준 코드.
* **Week Period (`base_week_period`)**: 기준 연/주차(YYYYWK)와 시작일/종료일, 기준 월(Month)을 매핑하여 타임라인 산정을 보조합니다.

## 5. 데이터 무결성 및 시스템 제약 (Integrity & Constraints)

### 5.1 고유 제약 조건 (Unique Constraints)
데이터의 논리적 중복을 DB 레벨에서 완벽히 차단하기 위해 `Meta.constraints` (UniqueConstraint)가 설정되어 있습니다.
* **예시 (Cascading):** 동일 시나리오 + 동일 Proforma 내에서 `vessel_position`(**포지션** 번호)은 중복될 수 없습니다. `["scenario", "proforma", "vessel_position"]`
* **방어 로직:** 서비스 레이어(`_crud_base.py`, `cascading_service.py` 등)에서 데이터 삽입 전 `exists()`를 체크하거나 `update_or_create()`를 통해 충돌을 제어합니다.

### 5.2 마스터 데이터 참조 무결성 (PROTECT) 및 삭제 경고
* 시나리오 데이터(`Sce_`)는 기준 데이터(`MasterLane`, `MasterPort` 등)를 참조할 때 `ON_DELETE=PROTECT`를 가집니다. 이는 사용 중인 노선 코드가 실수로 마스터에서 삭제되는 것을 방지합니다.
* **[주의]** `python manage.py init_master_data` 커맨드를 통해 마스터 테이블을 강제로 재초기화할 경우, 참조 무결성을 위해 하위의 모든 `sce_` 데이터와 `base_` 데이터가 **연쇄 삭제(CASCADE)** 되므로 시스템 초기화 시에만 사용해야 합니다.

### 5.3 DB 메타데이터 현행화 (DB Comments)
* Django ORM은 기본적으로 DB 컬럼 코멘트를 지원하지 않지만, `apps.py`의 `post_migrate` 시그널을 활용하여 모델의 `verbose_name`을 **PostgreSQL의 `COMMENT ON COLUMN` 쿼리로 자동 주입**합니다.
* 이를 통해 DBeaver 등 외부 툴로 DB에 직접 접근하는 데이터 엔지니어도 스키마의 의미를 즉각적으로 파악할 수 있습니다.
