# 2. 데이터베이스 설계 및 모델링 가이드 (Database Design & Modeling Guide)

## 1. 개요 (Overview)
본 시스템은 시뮬레이션의 기준이 되는 **마스터 데이터(Base Data)**와 시뮬레이션별로 독립적으로 조작 가능한 **시나리오 데이터(Scenario Data)**를 엄격히 분리하여 관리합니다. 
Django의 `Abstract Base Class(추상 모델)` 패턴을 적용하여 데이터 구조의 일관성을 보장하며, 복잡한 해운 물류 데이터를 효율적으로 처리하기 위해 고도의 **정규화(Normalization)** 및 **계층화** 설계를 적용했습니다.

---

## 2. 데이터 모델링 원칙 (Modeling Principles)

### 2.1 3단계 추상 모델 패턴 (3-Tier Abstract Model Pattern)
핵심 비즈니스 로직 테이블은 중복 코드를 방지하기 위해 3단계 상속 구조를 가집니다.
1. **Abstract Model (`Abs...`)**: 실제 DB 테이블을 생성하지 않으며, 필드 정의만 담당합니다.
2. **Base Model (`Base...`)**: 기준 정보(Master Data)를 저장하는 테이블입니다. (Scenario FK 없음)
3. **Scenario Model (Prefix 없음)**: 특정 시나리오(`ScenarioInfo`)에 종속된 데이터 저장소입니다.

### 2.2 테이블 명명 규칙 (Naming Convention)

| 모델 클래스명 | DB 테이블명 | 역할 및 특징 |
| :--- | :--- | :--- |
| `AbsVesselInfo` | (None) | 필드 정의 (상속용 추상 클래스) |
| `BaseVesselInfo` | `base_vessel_info` | **[Master]** 전사 공통 기준 데이터 (Scenario FK 없음) |
| `VesselInfo` | `sce_vessel_info` | **[Input]** 시나리오별 시뮬레이션 입력 데이터 (Scenario FK 포함) |
| `SimulationRun` | `sce_simulation_run` | **[Result]** 시뮬레이션 실행 기록 및 파라미터 내역 |

---

## 3. 공통 모델 정의 (Common Models)
모든 테이블에 공통으로 적용되는 감사(Audit) 로그 및 관계 설정을 위한 부모 모델입니다.

### 3.1 CommonModel (`common/models.py`)
데이터의 생성/수정 이력을 추적하기 위한 추상 모델입니다.
* `created_at` / `updated_at`: 생성 및 수정 일시 (Auto Now)
* `created_by` / `updated_by`: 생성자 및 수정자 (User FK)

### 3.2 ScenarioBaseModel (`input_data/models.py`)
시나리오 파생 데이터(`Sce_`)가 반드시 가져야 할 외래 키(FK)를 정의합니다.
* `scenario`: `ScenarioInfo` 모델 참조 (`ON_DELETE=CASCADE`)
* **특징:** 부모 시나리오 삭제 시, 하위 수십 개 테이블의 데이터가 DB 레벨에서 즉시 연쇄 삭제(Cascade)되어 고아 데이터를 방지합니다.

## 4. 논리적 데이터 그룹 (Logical Data Groups)

### 4.1 Scenario Master (Root) & Analysis
모든 시나리오 데이터의 뿌리가 되는 메타 정보 및 분석 결과 그룹입니다.
* **ScenarioInfo (`sce_scenario_info`)**: `code` (SCYYYYMMDD_NNN 자동 생성 고유 코드), `base_scenario` (복제 원본 추적 FK), `base_year_week`, `planning_horizon_months` 관리.
* **SimulationRun (`sce_simulation_run`)**: 최적화 엔진 실행 이력, 파라미터 및 요약 결과.
* **KPISnapshot (`sce_kpi_snapshot`)**: 시나리오별 주요 지표(Total TEU, Total Vessels, Utilization 등) 이력 보관.
* **ScenarioComparison (`sce_scenario_comparison`)**: 여러 시나리오 간의 비교 분석 결과 맵핑.

### 4.2 Group 1: Schedule & Cascading [★핵심]

#### **Proforma Schedule (`schedule_proforma` / `schedule_proforma_detail`)**
* **Master Header (`sce_proforma_schedule`)**
  * `lane`: FK to MasterLane
  * `proforma_name`: Proforma 이름 (4자리 숫자)
  * `effective_from_date` / `effective_to_date`: 적용 기간
  * `duration`: 전체 항해 기간 (일) - 첫 기항지 ETB에서 마지막 기항지 ETB까지
  * `declared_capacity`: 선언된 적재 용량 (TEU)
  * `declared_count`: 선언된 필요 선박 수
  * `own_vessel_count`: 배선으로 배치된 자선 수
  * 특징: 여러 기항지(Detail)를 포함하는 마스터 정보

* **Detail Lines (`sce_proforma_schedule_detail`)**
  * `proforma`: FK to ProformaSchedule (Master)
  * `port`: FK to MasterPort (기항지)
  * `calling_port_seq`: 기항지 순서 번호
  * `calling_port_indicator`: 기항지 호출 표시자
  * `turn_port_info_code`: Turning Port 여부 (Y/N) - Virtual Port 생성 여부 결정
  * `pilot_in_hours`: 외항에서 부두까지 소요 시간
  * `etb_day_number`: 첫 기항지 ETB로부터 경과 일수
  * `etb_day_code` / `etb_day_time`: ETB 요일 및 시간
  * `actual_work_hours`: 부두 작업 시간
  * `etd_day_number` / `etd_day_time`: ETD 정보
  * 특징: Excel 업로드 시에는 사용자가 1개 Grid로 입력하지만, DB 저장 시 Header/Detail로 정규화

#### **Lane Proforma Mapping (`lane_proforma_mapping`)**
* 역할: 특정 노선(Lane)에 어떤 Proforma를 언제부터 언제까지 적용할지 타임라인 기반 매핑
* 필드: `lane` (FK), `proforma` (FK), `effective_from_date`, `effective_to_date`
* 용도: Proforma 적용 시기를 유연하게 관리 (계절별 다른 스케줄 적용 등)

#### **Cascading Schedule (`schedule_cascading`)**
* 역할: Proforma의 특정 **포지션(Position)** 번호들을 미리 선택하고 선박 투입을 위해 확보하는 스케줄
* 필드: `proforma` (FK), `vessel_position` (포지션 번호, 1~declared_count)
* 특징: 아직 선박 코드가 지정되지 않음 - 다음 단계(CascadingVesselPosition)에서 구체적인 선박 배치
* 고유 제약: (scenario, proforma, vessel_position) 조합은 중복 불가

#### **Cascading Vessel Position (`schedule_cascading_vessel_position`)**
* 역할: Cascading Schedule의 포지션에 실제 선박을 배치하고 투입 시작일 지정
* 필드:
  * `cascading_schedule`: FK (위 Cascading Schedule 참조)
  * `vessel_code`: FK to VesselInfo (실제 선박)
  * `vessel_position_date`: 선박 투입 시작 일자
  * `own_type`: O (Own) / C (Chartered)
* 특징: 동일 포지션에 여러 선박을 시간차를 두고 배치 가능 (선박 교체 시나리오)

#### **Long Range Schedule (`schedule_long_range`)**
* 역할: 최적화 엔진의 최종 입력값이 되는 장기 운항 계획
* 프로세스:
  1. CascadingVesselPosition으로부터 선박 및 투입 시작일 획득
  2. Proforma 기간(Duration)을 반복 적용하여 항차(Voyage) 루프
  3. 각 항차별 기항지, 입출항 시간 자동 계산
  4. Turning Port(Y) 조건에 따라 Virtual Port 자동 생성
     * Virtual Port: 이전 항차 마지막 기항지와 현재 항차 첫 기항지 사이 개념적 포트
     * 엔진이 항차 전환을 정확히 추적하기 위해 필요

* 필드: `vessel` (FK), `voyage_number`, `etb`, `etd`, `port` (FK), `port_seq`, `is_virtual_port` 등

#### **기타 Schedule 관련 모델**
* **WeekPeriod (`base_week_period`)**: 
  * YYYYWK → 시작일/종료일 매핑 테이블
  * 타임라인 계산 및 주차 기반 쿼리 성능 최적화

### 4.3 Group 2: Vessel (선박)

* **Vessel Info (`base_vessel_info` / `sce_vessel_info`)**
  * `vessel_code`: 선박 고유 코드 (Primary Key)
  * `vessel_name`: 선박명
  * `own_type`: O (Own) / C (Chartered)
  * `vessel_size_description`: 선박 크기 (Panamax, New Panamax 등)
  * `delivery_date` / `redelivery_date`: 자선/용선 인도/반선 일자
  * `vessel_status`: ACTIVE, INACTIVE, UNDER_MAINTENANCE 등
  * 역할: 선박의 기본 제원 정보

* **Charter Cost (`base_vessel_charter_cost` / `sce_vessel_charter_cost`)**
  * `vessel`: FK to VesselInfo
  * `cost_type`: 용선료 유형 (Time Charter, Voyage Charter 등)
  * `start_date` / `end_date`: 적용 기간
  * `hire_rate`: 일일 용선료 ($/일)
  * 역할: 시간 기반 용선료 계산

* **Vessel Capacity (`base_vessel_capacity` / `sce_vessel_capacity`)**
  * `vessel`: FK to VesselInfo
  * `lane`: FK to MasterLane
  * `direction`: E (East) / W (West) / S (South) / N (North)
  * `capacity_teu`: 적재 가능량 (TEU)
  * 역할: 노선/방향별 실제 적재 용량 (declared_capacity와 상이할 수 있음)

### 4.4 Group 3: Cost & Distance (비용 및 거리)

* **Distance (`base_cost_distance` / `sce_cost_distance`)**
  * `from_port` / `to_port`: FK to MasterPort
  * `distance_nm`: 표준 운항 거리 (해리)
  * `eca_distance_nm`: ECA(배출통제해역) 구간 거리
  * `average_speed_knots`: 참고 평균 선속 (노트)
  * 역할: 항구 간 거리 및 시간 계산 기반 데이터

* **Canal Fee (`base_cost_canal_fee` / `sce_cost_canal_fee`)**
  * `canal_name`: 운하명 (Suez, Panama 등)
  * `from_port` / `to_port`: 운하 통과 경로
  * `direction`: E / W
  * `fee_per_teu`: TEU당 통행료
  * `fee_fixed`: 고정 통행료
  * 역할: 운하 통행료 계산

* **TS Cost (`base_cost_ts_cost` / `sce_cost_ts_cost`)**
  * `ts_port`: FK to MasterPort (환적 항만)
  * `cost_per_teu`: 환적 비용
  * `cost_per_container`: Container 당 비용
  * 역할: 환적(Transshipment) 항만 비용

### 4.5 Group 4: Bunker (연료)

* **Bunker Consumption (`base_bunker_consumption_sea` / `sce_bunker_consumption_sea`)**
  * `vessel_size`: Panamax, New Panamax 등 선박 크기 분류
  * `speed_knots`: 선속 (노트)
  * `consumption_ton_per_day`: 일일 소비량 (톤)
  * 역할: 선속별 해상 항해 중 연료 소비량

* **Bunker Consumption Port (`base_bunker_consumption_port` / `sce_bunker_consumption_port`)**
  * `vessel_size`: 선박 크기
  * `port_idle_consumption_ton_per_day`: 정박 중 아이들링 소비량
  * 역할: 항만 정박 중 연료 소비량

* **Bunker Price (`base_bunker_price` / `sce_bunker_price`)**
  * `bunker_type`: LSFO (Low Sulphur Fuel Oil) / MGO (Marine Gas Oil)
  * `region`: 지역/항로 (Singapore, Rotterdam 등)
  * `effective_date`: 가격 적용 시작일
  * `price_per_ton`: 톤당 가격 ($/톤)
  * 역할: 연료 유형/지역별 단가 관리

### 4.6 Group 5: Constraints (제약 조건)

* **Fixed Deployment (`constraint_fixed_deployment`)**
  * `lane`: FK to MasterLane
  * `vessel`: FK to VesselInfo (또는 NULL - 전체 제약)
  * `deployment_type`: I (Include, 필수 투입) / E (Exclude, 금지)
  * `effective_from_date` / `effective_to_date`: 적용 기간
  * 역할: 특정 노선에 특정 선박의 투입 필수/금지 제약

* **Fixed Schedule Change (`constraint_fixed_schedule_change`)**
  * `vessel`: FK to VesselInfo
  * `event_type`: I (Phase In) / O (Phase Out) / S (Schedule Change) / R (Port Swap) / L (Vessel Slide)
  * `event_date`: 이벤트 발생 일자
  * `affected_port`: 영향을 받는 기항지 (Optional)
  * `event_description`: 상세 설명
  * 역할: 선박의 투입/철수, 일정 변경, 항만 스킵 등 고정 이벤트

* **Port Constraint (`constraint_port`)**
  * `port`: FK to MasterPort
  * `max_vessel_size`: 입항 가능 최대 선박 크기
  * `draft_limitation_meter`: 최대 흘수 제한
  * `other_constraints`: JSON 형식의 추가 제약 (예: 특정 요일 입항 불가 등)
  * 역할: 항만별 물리적/운영 제약 조건

### 4.7 Group 6: Master Data (불변 기준 정보)

* **MasterTrade (`master_trade`)**
  * `trade_code`: Trade Code (Primary Key)
  * `trade_name`: Trade명 (예: AEX, TAS, FAX)
  * `from_continent_code` / `to_continent_code`: 무역로 방향
  * 역할: 전사 공통 무역로 마스터 데이터

* **MasterPort (`master_port`)**
  * `port_code`: Port Code (Primary Key, 예: KRPUS - 부산)
  * `port_name`: 항구명
  * `continent_code` / `country_code`: 지역 분류
  * 역할: 전사 공통 항구 마스터 데이터

* **MasterLane (`master_lane`)**
  * `lane_code`: Lane Code (Primary Key, 예: AEX)
  * `lane_name`: Lane 이름
  * `vessel_service_type_code`: 서비스 유형 코드
  * `main_trade`: FK to MasterTrade (주요 무역로)
  * 역할: 전사 공통 노선 마스터 데이터

* **Week Period (`base_week_period`)**
  * `year_week`: YYYYWK 형식 (예: 202610)
  * `start_date` / `end_date`: 주차의 시작/종료 일자
  * `month`: 해당 월 (YYYYMM)
  * 역할: 주차 기반 타임라인 계산 및 쿼리 성능 최적화

## 5. 데이터 무결성 및 시스템 제약 (Integrity & Constraints)

### 5.1 고유 제약 조건 (Unique Constraints)

데이터의 논리적 중복을 DB 레벨에서 완벽히 차단하기 위해 `Meta.constraints` (UniqueConstraint)가 설정되어 있습니다.

* **예시 1 (Cascading Vessel Position):** 
  * 제약: `(scenario, cascading_schedule, vessel_code, vessel_position_date)` 조합 중복 불가
  * 의미: 동일 시나리오의 동일 포지션에는 동일 선박이 동일 시작일로 배치될 수 없음
  * 방어 로직: Service Layer에서 `update_or_create()` 사용 (중복 시 갱신)

* **예시 2 (Proforma Schedule):**
  * 제약: `(scenario, lane, proforma_name, direction)` 조합 중복 불가
  * 의미: 동일 시나리오의 동일 노선에는 동일 이름의 Proforma가 동일 방향으로 존재할 수 없음

* **예시 3 (Week Period):**
  * 제약: `(year_week)` 은 Primary Key (auto-increment 아님)
  * 의미: 연/주차는 하나만 존재

* **방어 전략:**
  * DB 레벨: `UNIQUE INDEX` 자동 생성
  * ORM 레벨: `IntegrityError` 발생 시 의미 있는 에러 메시지 반환
  * Service 레벨: `exists()` 체크 또는 `update_or_create()` 사용

### 5.2 마스터 데이터 참조 무결성 (ON_DELETE=PROTECT vs CASCADE)

* **PROTECT 정책 (마스터 보호):**
  * 시나리오 데이터(`Sce_`)가 마스터 데이터(`MasterLane`, `MasterPort` 등)를 참조할 때 적용
  * 효과: 사용 중인 마스터 코드가 실수로 삭제되는 것을 방지
  * 삭제 시도 시: `ProtectedError` 발생으로 삭제 거부
  * 예: KRPUS 항구를 사용 중인 Proforma가 있으면 MasterPort 삭제 불가

* **CASCADE 정책 (시나리오 정리):**
  * 부모 시나리오(`ScenarioInfo`) 삭제 시 하위의 모든 `Sce_` 테이블 데이터 자동 삭제
  * 효과: 시나리오 삭제 시 고아 데이터(Orphan Data) 자동 정리
  * 주의: 시나리오 삭제는 신중하게 (복원 불가)

* **[중요] Master Data 재초기화 시 주의:**
  * 커맨드: `python manage.py init_master_data --force`
  * 효과: Master 테이블 갱신 시 참조 무결성에 의해 하위의 모든 `sce_` 및 `base_` 데이터 연쇄 삭제(CASCADE)
  * 사용 시기: 시스템 초기 구성 또는 전체 초기화 시에만 사용
  * 안전장치: 커맨드 실행 시 확인 프롬프트 표시 (--force 옵션 없으면 실행 안 됨)

### 5.3 DB 메타데이터 현행화 (DB Comments)

* **문제:** Django ORM은 기본적으로 DB 컬럼 코멘트를 자동 생성하지 않음
* **해결책:** `input_data/apps.py`의 `post_migrate` 시그널을 활용
  * Django 모델의 `verbose_name` → PostgreSQL `COMMENT ON COLUMN` 쿼리로 자동 변환
  * 마이그레이션 실행 후 자동으로 DB 코멘트 설정

* **이점:**
  * DBeaver, pgAdmin 등 외부 DB 도구로 스키마 조회 시 필드의 의미를 즉각 파악 가능
  * 데이터 엔지니어의 별도 문서 작성 불필요
  * DB <-> ORM 간 메타데이터 동기화

* **예시:**
  ```sql
  COMMENT ON COLUMN sce_proforma_schedule.duration 
  IS '첫 기항지 ETB에서 마지막 기항지 ETB까지의 총 항해 기간 (일)';
  ```

### 5.4 Audit Trail & Change Tracking

* **django-simple-history 활용:**
  * `CommonModel` 상속 모델들의 모든 변경이 자동 기록됨
  * `created_by` / `updated_by` 필드로 변경자 추적
  * 각 변경마다 타임스탬프 기록
  * 이력 조회: `모델.history.all()`

* **로깅:**
  * 중요 비즈니스 로직(Cascading, LRS 생성 등)에서 `logging` 모듈 사용
  * 변경 내용, 실행 시간, 영향받은 레코드 수 기록
