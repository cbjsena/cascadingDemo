# **2\. 데이터베이스 설계 및 모델링 가이드 (Database Design & Modeling Guide)**

## **1\. 개요 (Overview)**

본 시스템은 시뮬레이션의 기준이 되는 \*\*마스터 데이터(Base Data)\*\*와 시뮬레이션별로 독립적으로 조작 가능한 \*\*시나리오 데이터(Scenario Data)\*\*를 엄격히 분리하여 관리합니다.

이를 위해 Django의 Abstract Base Class(추상 모델) 패턴을 적용하여 데이터 구조의 일관성을 보장하며, 복잡한 해운 물류 데이터를 효율적으로 처리하기 위해 고도의 **정규화(Normalization)** 및 **계층화** 설계를 적용했습니다.

## **2\. 데이터 모델링 원칙 (Modeling Principles)**

### **2.1 3단계 추상 모델 패턴 (3-Tier Abstract Model Pattern)**

모든 핵심 비즈니스 로직 테이블은 중복 코드를 방지하기 위해 3단계 상속 구조를 가집니다.

1. **Abstract Model (Abs...)**: 실제 DB 테이블을 생성하지 않으며, 필드(컬럼) 정의만 담당합니다.  
2. **Base Model (Base...)**: 기준 정보(Master Data)를 저장하는 테이블입니다. 시나리오 FK가 없습니다.  
3. **Scenario Model (Prefix 없음)**: 특정 시나리오(ScenarioInfo)에 종속된 데이터를 저장하는 테이블입니다.

### **2.2 테이블 명명 규칙 (Naming Convention)**

| 모델 클래스명 | DB 테이블명 | 역할 및 특징 |
| :---- | :---- | :---- |
| AbsVesselInfo | (None) | 필드 정의 (상속용 추상 클래스) |
| BaseVesselInfo | base\_vessel\_info | **\[Master\]** 전사 공통 기준 데이터 (Scenario FK 없음) |
| VesselInfo | sce\_vessel\_info | **\[Input\]** 시나리오별 시뮬레이션 입력 데이터 (Scenario FK 포함) |
| SimulationRun | sce\_simulation\_run | **\[Result\]** 시뮬레이션 실행 기록 및 결과 요약 (예정) |

## **3\. 공통 모델 정의 (Common Models)**

모든 테이블에 공통으로 적용되는 감사(Audit) 로그 및 관계 설정을 위한 부모 모델입니다.

### **3.1 CommonModel (common/models.py)**

데이터의 생성/수정 이력을 추적하기 위한 추상 모델입니다.

* created\_at / updated\_at: 생성 및 수정 일시 (Auto Now)  
* created\_by / updated\_by: 생성자 및 수정자 (User FK)

### **3.2 ScenarioBaseModel (input\_data/models.py)**

시나리오 파생 데이터(Sce\_)가 반드시 가져야 할 외래 키(FK)를 정의합니다.

* scenario: ScenarioInfo 모델 참조 (ON\_DELETE=CASCADE)  
* **특징:** 부모 시나리오 삭제 시, 해당 시나리오에 속한 모든 하위 데이터가 DB 레벨에서 즉시 삭제(Cascade)되어 고아 데이터(Orphan Data)를 방지합니다.

## **4\. 논리적 데이터 그룹 (Logical Data Groups)**

### **4.1 Scenario Master (Root)**

모든 시나리오 데이터의 뿌리가 되는 메타 정보 테이블입니다.

* **테이블:** sce\_scenario\_info  
* **주요 필드:** \* code (SCYYYYMMDD\_NNN 포맷의 자동 생성 고유 코드)  
  * scenario\_type (BASELINE, WHAT\_IF, OPTIMIZATION 등)  
  * base\_scenario (복제 원본 시나리오 추적용 자기 참조 FK)  
  * base\_year\_week / planning\_horizon\_months (시뮬레이션 대상 기간)

### **4.2 Group 1: Schedule & Cascading \[★핵심 변경\]**

스케줄 관리의 유연성을 위해 헤더(Master)와 기항지(Detail)를 분리하고, 선박 배치(Cascading) 모델을 신설했습니다.

* **Proforma Schedule (schedule\_proforma / schedule\_proforma\_detail)**  
  * **Master:** 노선별 스케줄 헤더 정보 (기간, 선복량, 필요 선박 수 등).  
  * **Detail:** 개별 기항지 순서, 입출항 시간(ETB/ETD), 소요 시간(Sea Time), 거리.  
  * **특징:** Excel 업/다운로드 시 UI에서는 1개의 Grid로 보이지만, DB 저장 시(Service Layer) Header와 Detail로 완벽히 정규화되어 저장됩니다.  
* **Lane Proforma Mapping (lane\_proforma\_mapping)**  
  * 역할: 특정 노선(Lane)에 시기별로 어떤 Proforma를 적용할지 타임라인 기반으로 매핑.  
* **Cascading Vessel Position (schedule\_cascading\_vessel\_position)**  
  * 역할: Proforma에서 요구하는 슬롯(Position)에 실제 선박(vessel\_code)을 배치하고 투입일(vessel\_position\_date)을 매핑.  
* **Long Range Schedule (schedule\_long\_range)**  
  * 역할: 최적화 엔진의 최종 입력값이 되는 장기 운항 계획.  
  * 로직: Proforma와 Cascading 데이터를 결합하여 항차(Voyage)를 무한 루프 돌리며, Turning Port(Y) 조건에 따라 \*\*가상 포트(Virtual Port)\*\*를 자동 파생 생성합니다.

### **4.3 Group 2: Vessel (선박)**

* **Vessel Info (vessel\_info)**: 선박 마스터 정보 (소유 구분, 인도/반선일, 수리 일정).  
* **Charter Cost (vessel\_charter\_cost)**: 용선료 정보 (기간별 hire\_rate).  
* **Vessel Capacity (vessel\_capacity)**: 항로/선박/방향별 적재 가능량(vessel\_capacity, reefer\_capacity).

### **4.4 Group 3: Cost & Distance (비용 및 거리)**

* **Distance (cost\_distance)**: 항구 간 표준 운항 거리 및 ECA(배출통제해역) 거리.  
* **Canal Fee (cost\_canal\_fee)**: 운하(Suez, Panama 등) 통과 방향별 비용.  
* **TS Cost (cost\_ts\_cost)**: 환적(Transshipment) 항만 비용.

### **4.5 Group 4: Bunker (연료)**

* **Bunker Consumption (bunker\_consumption\_sea / \_port)**: 선박 크기 및 선속(Speed)에 따른 해상 소모량 및 정박/아이들링/도선 시 소모량.  
* **Bunker Price (bunker\_price)**: 지역/항로별 연료 단가 (LSFO, MGO).

### **4.6 Group 5: Constraints (제약 조건)**

* **Fixed Deployment (constraint\_fixed\_deployment)**: 특정 항로에 선박 투입 필수(Include) 또는 금지(Exclude) 제약.  
* **Fixed Schedule Change (constraint\_fixed\_schedule\_change)**: Phase-In/Out, Port Omission 등 고정 이벤트.  
* **Port Constraint (constraint\_port)**: 항구(터미널)별 입항 가능한 최대 선박 크기 제한.

## **5\. 데이터 무결성 및 시스템 제약 (Integrity & Constraints)**

### **5.1 고유 제약 조건 (Unique Constraints)**

데이터의 논리적 중복을 DB 레벨에서 완벽히 차단하기 위해 Meta.constraints (UniqueConstraint)가 설정되어 있습니다.

* **예시 (Cascading):** 동일 시나리오 \+ 동일 Proforma 내에서 vessel\_position(슬롯 번호)은 중복될 수 없습니다. \["scenario", "proforma", "vessel\_position"\]  
* **방어 로직:** 서비스 레이어(\_crud\_base.py, cascading\_service.py 등)에서 데이터 삽입 전 exists()를 체크하거나 update\_or\_create()를 통해 충돌을 제어합니다.

### **5.2 마스터 데이터 참조 무결성 (PROTECT)**

* 시나리오 데이터(Sce\_)는 마스터 기준 데이터(MasterLane, MasterPort 등)를 참조할 때 ON\_DELETE=PROTECT 제약을 가집니다.  
* 이는 이미 시뮬레이션에 사용 중인 항구나 노선 코드가 마스터 테이블에서 실수로 삭제되는 것을 DB 레벨에서 강력히 방지합니다.

### **5.3 DB 메타데이터 현행화 (DB Comments)**

* Django ORM은 기본적으로 DB 컬럼 코멘트를 지원하지 않지만, apps.py의 post\_migrate 시그널을 활용하여 모델의 verbose\_name을 **PostgreSQL의 COMMENT ON COLUMN 쿼리로 자동 주입**합니다.  
* 이를 통해 DBeaver 등 외부 툴로 DB에 직접 접근하는 데이터 엔지니어도 스키마의 의미를 즉각적으로 파악할 수 있습니다.
