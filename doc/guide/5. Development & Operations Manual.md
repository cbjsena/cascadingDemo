# **5\. 개발 환경 및 운영 매뉴얼 (Development & Operations Manual)**

## **1\. 개요 (Overview)**

본 문서는 Cascading Opt 시스템의 로컬 개발 환경 구성(Local Setup), Docker 기반 실행, 초기 데이터 적재 자동화, 테스트 환경 및 트러블슈팅 가이드를 제공합니다.

### **1.1 시스템 요구사항 (Prerequisites)**

* **OS:** Windows 10/11 (WSL2 권장), macOS, Linux  
* **Language:** Python 3.11+  
* **Database:** PostgreSQL 14+ (로컬 테스트 및 운영 환경 필수)  
* **Cache/Queue:** Redis 7+  
* **Tool:** Docker Desktop, Git

## **2\. 로컬 개발 환경 구성 (Local Development Setup)**

### **2.1 프로젝트 클론 및 가상환경 설정**

\# 1\. Repository Clone  
git clone \<repository\_url\>  
cd cascadingDemo

\# 2\. Virtual Environment 생성 (Python 3.11)  
python \-m venv venv

\# 3\. Virtual Environment 활성화  
\# Windows:  
venv\\Scripts\\activate

source venv/bin/activate

### **2.2 의존성 패키지 설치**

\# 패키지 설치  
pip install \--upgrade pip  
pip install \-r requirements.txt

\# Pre-commit Hook 설치 (코드 품질 자동 관리)  
pre-commit install

### **2.3 환경 변수 설정 (.env)**

프로젝트 루트의 .env.local 파일을 복사하여 .env로 이름을 변경하고, 데이터베이스 접속 정보(DB\_HOST, DB\_USER, DB\_PASSWORD)를 로컬 환경에 맞게 수정합니다.

### **2.4 데이터베이스 초기화 및 서버 실행**

python manage.py runserver

*(참고: 마이그레이션이 완료되면 post\_migrate 시그널에 의해 슈퍼유저 생성 및 초기 데이터 적재가 자동으로 수행됩니다. 상세 내용은 4장 참조)*

## **3\. Docker 기반 실행 (Docker Deployment)**

docker-compose를 사용하여 DB, Redis, Web, Worker 컨테이너를 한 번에 띄우는 배포 환경 시뮬레이션입니다.

\# 백그라운드 모드 실행  
docker-compose up \-d \--build

\# 실행 상태 확인  
docker-compose ps

\# Web 컨테이너 내부에서 마이그레이션 실행  
docker-compose exec web python manage.py migrate

## **4\. 초기 데이터 적재 및 시스템 자동화 (Auto-Setup)**

본 시스템은 운영 편의성을 위해 DB 마이그레이션(`python manage.py migrate`) 실행 시 `input_data/apps.py`의 `post_migrate` 시그널에 의해 시스템 초기화 스크립트가 백그라운드에서 자동 실행됩니다.

### **4.1 자동화 상세 프로세스 (input\_data/apps.py)**

1. **DB 메타데이터 동기화**: `update_db_comments` 커맨드가 실행되어 Django 모델의 `verbose_name`을 PostgreSQL `COMMENT ON COLUMN` 쿼리로 자동 주입합니다.  
2. **테이블 정의서 생성**:`generate_table_definition` 커맨드가 실행되어 DB 시스템 카탈로그를 바탕으로 `doc/db/` 폴더에 `base_table_definitions.csv` 문서를 자동 갱신합니다.  
3. **슈퍼유저 자동 생성:** `create_default_superuser` 커맨드가 실행되어 cascading / qwer123$ 계정이 존재하지 않으면 자동 생성한다.
4. **마스터 및 기준 데이터 적재 (`_base_loader.py`)**: 
  * `init_master_data`: 실행 시 기존 데이터가 연쇄 삭제(CASCADE)될 수 있으므로 주의가 필요하며, `--force` 옵션으로 강제 덮어쓰기가 가능합니다.
  * `init_base_data`: 대량의 벙커/비용 데이터 적재 시 `bulk_create(ignore_conflicts=True)`를 사용하여 중복 에러를 무시하고 적재 속도를 극대화했습니다.

### **4.2 수동 데이터 관리 스크립트 (management/commands)**

CSV 데이터가 변경되어 수동으로 다시 적재해야 할 때 사용합니다. 공통 기능은 `_base_loader.py`에 구현되어 대량 Insert(bulk_create) 및 Lazy Loading 캐시를 통해 고속 처리됩니다.

* **python manage.py init_base_data**: 시나리오나 마스터 데이터를 건드리지 않고 base_ 테이블만 갱신합니다.  
* **python manage.py init_master_data**: master_ 테이블을 갱신합니다.  
  * **[주의]** master_ 테이블 갱신 시 외래키 무결성을 위해 sce_ (시나리오) 데이터와 base_ 데이터가 **연쇄 삭제(CASCADE)** 됩니다. 실행 시 방어 프롬프트가 뜨며, 자동화 환경에서는 --force 옵션을 사용합니다.

## **5. 테스트 및 코드 품질 관리 (Testing & QA)**

프로젝트는 최상위 폴더인 `tests/` 구조를 채택하여 서비스(Production) 로직과 테스트(Test) 환경을 완벽히 분리했습니다.

* **BDD (Behavior-Driven Development)**: 기획 단계에서 도출된 방대한 CSV 기반 테스트 시나리오(`test_scenarios_api.csv`, `test_scenarios_input_data.csv`)의 **Given-When-Then** 명세를 준수하여 약 190여 개의 테스트가 작성되었습니다.
* **공통 픽스처 (`conftest.py`)**: 사용자 인증 정보(Client), 기본 시나리오, 마스터 노선/항구 정보 등이 전역에 Mocking 및 의존성 주입되어 테스트 속도와 격리성을 완벽히 보장합니다.

### **5.3 주요 검증 포인트 (Highlights)**

* Master CSV I/O 통합 테스트 (test_view_master.py): 마스터 팩토리의 CSV 다운로드 시 인코딩(BOM) 정상 적용 여부, 그리고 가상의 CSV 파일 스트림(io.BytesIO)을 통한 업로드 기능 및 예외 처리 방어 로직이 완벽하게 자동화 테스트로 구현되어 있습니다.
* 스케줄링 역산 방어 (test_service_proforma.py): ETB를 이전 ETD보다 과거로 입력했을 때, 시스템이 다음 주차 요일로 정확히 밀어내는지(_resolve_next_etb) 검증합니다.
* 장기 운항 계획 파생 (test_service_lrs.py): Turn Port = Y인 경우 반대 방향(Opposite Direction)으로 가상 포트가 의도대로 분리 생성되는지 검증합니다.

## **6. 운영 트러블슈팅 (Troubleshooting)**

**Q1. DB 마이그레이션 충돌 발생 (DuplicateTable 등)**

* **해결:** input_data/migrations/ 폴더 내 `__init__.py`를 제외한 모든 파일을 삭제하고, DB 테이블을 Drop 한 뒤 `makemigrations` -> `migrate`를 재수행합니다.

**Q2. CSV 데이터 로드 시 한글 깨짐 오류**

* **해결:** 엑셀에서 저장한 CSV가 cp949로 저장되었을 가능성이 높습니다. 에디터로 열어 **UTF-8 with BOM (utf-8-sig)** 형식으로 다시 저장한 후 적재 스크립트를 재실행합니다.

**Q3. `test_*.py` 실행 시 FileNotFoundError 발생 (CSV 파일)**

* **원인:** 테스트 폴더 구조가 루트 tests/로 이동하면서 테스트 데이터(data 폴더 내 CSV) 경로를 찾지 못함.  
* **해결:** 절대경로 하드코딩 대신 os.path.dirname(os.path.abspath(__file__))을 사용하여 현재 테스트 파일 위치를 기준으로 한 동적 상대 경로로 수정해야 합니다.

**Q4. Celery Worker가 동작하지 않아 시나리오 복제가 멈춤**

* **해결:** docker ps로 Redis 컨테이너 동작 여부를 확인하고, .env의 CELERY_BROKER_URL을 확인합니다. 로컬 개발 및 디버깅 시에는 CELERY_TASK_ALWAYS_EAGER = True를 설정하여 비동기 작업을 동기로 실행할 수 있습니다.
