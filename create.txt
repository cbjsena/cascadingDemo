7. 서버 실행

    7.2 테스트 실행
        - pytest run/tests/ -vv -rw
        - pytest run/tests/test_run_views.py
        - pytest run/tests/test_run_models.py::RunExecutionModelTest            // 특정 클래스의 모든 테스트 실행
        - pytest run/tests/test_run_views.py::test_post_creates_full_snapshot   //특정 함수(테스트 케이스)만 실행
        - pytest -m snapshot //마커가 "snapshot"인 테스트만 실행 (@pytest.mark.snapshot, 유형별 테스트 그룹 관리에 유용)
    7.3 pytest 옵션
        -v: verbose
        -vv: very verbose, 어떤 부분에서 경고가 발생했는지 줄번호까지 표시
        -x: 실패 시 즉시 중단
        --maxfail=N: N회 실패 시 중단
        --lf: 마지막 실패한 테스트들만 다시 실행
        -k "run": "run"이란 단어가 포함된 테스트만 실행
        -m MARKEXPR: 특정 마커가 있는 테스트만 실행
        --ignore: 특정 파일/디렉토리무시
        --W ignore::DeprecationWarning: DeprecationWarning 무시
        -rw: 요약 정보 출력 (통과, 실패, 건너뜀 등)
        -rw > warnings.log: 경고 요약 정보를 warnings.log 파일로 리다이렉트
        --tb=short: 짧은 트레이스백 출력
        --log-cli-level: 출력 로그 레벨 설정 (예: DEBUG, INFO)
        --cache-clear: pytest 캐시 초기화, 테스트를 반복 실행하면서 이상한 문제가 생겼을 때 유용

    8.5 vscode 자동화 testing

9. git
    9.1 tag
        - 태그 목적
            - 특정 버전 스냅샷을 표시
            - 배포 시점 표시
            - 롤백을 쉽게 하기 위해
            - Release를 만들기 위한 기반
        - tag 생성: git tag -a v0.1 -m "Initial release"
        - tag 푸시: git push origin v0.1
        - tag 삭제: git tag -d v0.1
        - 원격 저장소에서 tag 삭제: git push origin --delete v0.1
        - Lightweight Tag: git tag v0.2
        - tag 목록 보기: git tag
        - 특정 tag 정보 보기: git show v0.1
    9.2 stash
        - 작업 도중에 변경 사항을 임시로 저장하고 나중에 다시 적용할 때 사용
        - Git이 추적(tracked) 중인 파일만 저장
        - 변경 사항 임시 저장: git stash
        - 특정 파일만 임시 저장: git stash push <파일명>
        - 임시 저장된 변경 사항 목록 보기: git stash list
        - 가장 최근에 임시 저장된 변경 사항 적용: git stash apply
        - 특정 stash 적용: git stash apply stash@{1}
        - 임시 저장된 변경 사항 적용 후 삭제: git stash pop
        - 특정 stash 삭제: git stash drop stash@{1}
        - 모든 stash 삭제: git stash clear
    9.3 rebase
        - 브랜치의 커밋 히스토리를 재정렬하거나 변경할 때 사용
        - 현재 브랜치를 다른 브랜치 위로 이동: git checkout feature-branch && git rebase main
        - 충돌 해결 후 계속 진행: git add <충돌해결된파일> && git rebase --continue
        - rebase 중단: git rebase --abort
        - 대화형 rebase: git rebase -i HEAD~3  //최근 3개의 커밋을 편집 모드로 열기
    9.4 cherry-pick
        - 다른 브랜치의 특정 커밋을 현재 브랜치에 적용할 때 사용
        - 커밋 해시 확인: git log --oneline
        - cherry-pick 적용: git cherry-pick <커밋해시>
        - 충돌 해결 후 계속 진행: git add <충돌해결된파일> && git cherry-pick --continue
        - cherry-pick 중단: git cherry-pick --abort

11. 배포
    11.1 static 개발 서버 vs 배포 서버 동작 차이
        - DEBUG = True (runserver 사용)일 때는 django.contrib.staticfiles가 정적 파일을 자동으로 서빙
        - DEBUG = False (실서버)면 Django가 직접 서빙하지 않으므로:
            - python manage.py collectstatic로 모든 static 파일을 STATIC_ROOT로 모으고(예: staticfiles/),
            - 웹서버(nginx, Apache)나 WhiteNoise로 정적 파일을 제공
            - 개발 환경에서 DEBUG=False이면 파일이 안 보일 수 있으니 DEBUG 값을 확인
        - collectstatic 테스트

14. 서버 배포
    gunicorn + nginx
        - gunicorn: 파이썬 WSGI HTTP 서버, Django 애플리케이션을 서비스하는 역할
        - nginx: 정적 파일 서빙 및 리버스 프록시 역할
    14.1 gunicorn 설치
        - pip install gunicorn
    14.2 Nginx 설정 파일 생성(./nginx/nginx.conf)
    14.3 시작 스크립트 생성(./scripts/entrypoint.sh)이나 정적 파일 수집
        - 컨테이너가 실행될 때 1) DB 마이그레이션, 2) 정적 파일 모으기(collectstatic), 3) Gunicorn 실행을 순서대로 처리하기 위한 스크립트
        - 스크립트 없다면 정적 파일 수집
            - python manage.py collectstatic
    14.4 dock-compose.yml 수정
    14.5 config/settings.py
        - DEBUG, ALLOWED_HOSTS를 env에서 가져오도록 수정
            - DEBUG = os.getenv("DEBUG", "False").lower() == "true"
            - ALLOWED_HOSTS = os.environ.get("ALLOWED_HOSTS", "localhost,127.0.0.1").split(",")
            - Nginx를 통해 들어오면 Host 헤더가 유지되므로, .env.docker 파일에 배포할 서버의 IP나 도메인을 추가
        - 정적 파일이 모일 경로 (Nginx가 이곳을 참조함)
            - STATIC_URL = '/static/'
            - STATIC_ROOT = os.path.join(BASE_DIR, 'staticfiles')
    14.6 배포
        - 기존 컨테이너 중지 및 삭제: podman-compose down (또는 docker-compose down)
        - 새로 빌드 및 실행: podman-compose up -d --build
    14.7 domain 설정(로컬)
        1) C:\Windows\System32\drivers\etc\hosts 파일 수정 (관리자 권한 필요)
            - 127.0.0.1 opusopt-demo.com
        2) Django 설정 수정 (.env.docker)
            - ALLOWED_HOSTS=localhost,127.0.0.1,opusopt-demo.com
            - CSRF_TRUSTED_ORIGINS=http://opusopt-demo.com # CSRF 보안 설정 (https가 아닌 http로 접속 시 필요)
        3) Nginx 설정 수정 (./nginx/nginx.conf)
            - server_name opusopt-demo.com;
        4) 배포 재실행
            - podman-compose down
            - podman-compose up -d --build
            - 빌드 시 rootlessport cannot expose privileged port 80 오류 발생 시:
                - docker-compose.yml의 nginx: 의 ports:를 "80:80" -> "8080:80" 등으로 변경
            - 포트 없이 접속하려면 80 포트로 접속해야 하므로 관리자 권한으로 rootlessport 설정 필요
                - cmd에서 wsl 쳐서 리눅스 접속
                - sudo sysctl -w net.ipv4.ip_unprivileged_port_start=80 # 1. 80번 포트부터 사용 가능하도록 설정
                - echo "net.ipv4.ip_unprivileged_port_start=80" | sudo tee -a /etc/sysctl.conf # 2. 재부팅 후에도 유지되도록 설정 추가
